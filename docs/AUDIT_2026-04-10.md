## CRITICOS (11) — Requieren accion inmediata

### C1. Build ROTO — `react-day-picker/locale` no resuelve
- **Archivo**: `src/components/shared/filter-date-range.tsx:12`
- **Impacto**: La aplicacion NO se puede desplegar
- **Fix**: Instalar dependencia faltante o cambiar import

### C2. Autenticacion sin password — TODO pendiente
- **Archivo**: `src/lib/auth/config.ts:61`
- **Impacto**: Cualquiera con un email valido puede autenticarse sin contrasena
- **Detalle**: `emailOnlySchema` solo valida email, `bcrypt.compare()` esta comentado con `// TODO: Re-enable password verification`
- **Fix**: Reactivar verificacion de contrasena con bcrypt

### C3. Middleware no protege todas las rutas del dashboard
- **Archivo**: `src/lib/auth/config.ts:17-24`
- **Impacto**: `/clients`, `/intercom`, `/warehouse`, `/analytics` accesibles sin autenticacion
- **Fix**: Anadir las rutas faltantes al callback `authorized` o cambiar a enfoque negativo (proteger todo excepto publico)

### C4. Webhook Intercom acepta requests sin firma HMAC
- **Archivo**: `src/app/api/webhooks/intercom/route.ts:110-124`
- **Impacto**: Un atacante puede insertar datos arbitrarios en `intercom_inbox`
- **Fix**: Requerir siempre `x-hub-signature-256`, rechazar con 401 si falta

### C5. Queries de lista NO filtran clientes/proveedores soft-deleted
- **Archivo**: `src/server/queries/incidents.ts:86-92`, `src/server/queries/rmas.ts:86-93`
- **Impacto**: Incidencias y RMAs muestran clientes/proveedores eliminados
- **Fix**: Anadir `isNull(clients.deletedAt)` / `isNull(providers.deletedAt)` en los JOINs

### C6. Alertas SLA no restan `slaPausedMs`
- **Archivo**: `src/server/queries/alerts.ts:124-129`
- **Impacto**: Falsos positivos de alertas SLA para incidencias pausadas
- **Fix**: Restar `slaPausedMs` del calculo elapsed, como ya se hace en `dashboard.ts`

### C7. Sin tests para server queries (0% cobertura)
- **Directorio**: `src/server/queries/` (15 archivos, 0 tests)
- **Impacto**: Logica critica de negocio sin verificacion automatica

### C8. Sin tests para 17 de 20 server actions (15% cobertura)
- **Directorio**: `src/server/actions/`
- **Impacto**: Solo incidents, rmas y users tienen tests

### C9. Tests RMA usan nombres de estado OBSOLETOS — suite invalida
- **Archivo**: `src/server/actions/rmas.test.ts`
- **Impacto**: Los tests de transiciones RMA usan `aprobado_proveedor`, `recibido_proveedor`, `en_reparacion_proveedor`, `recibido_almacen` — nombres que **NO EXISTEN** en constants ni state machine (los correctos son `aprobado`, `en_proveedor`, `recibido_oficina`)
- **Consecuencia**: Los tests pasan accidentalmente (Zod rechaza los estados invalidos, `result.success === false` es el outcome esperado en algunos casos). Toda la suite de lifecycle RMA da falsa confianza
- **Fix**: Actualizar todos los status names en `rmas.test.ts` a los correctos

### C10. Sin testing E2E
- **Impacto**: No hay Playwright ni Cypress; flujos criticos no se verifican end-to-end

### C11. Sin coverage thresholds en Vitest
- **Archivo**: `vitest.config.mts`
- **Impacto**: No hay umbrales de cobertura; la cobertura puede caer a 0% sin que CI falle
- **Fix**: Anadir `coverage: { thresholds: { lines: 70, functions: 70 } }` y instalar `@vitest/coverage-v8`

### C12. Paginacion Intercom sin accesibilidad
- **Archivo**: `src/components/intercom/intercom-inbox.tsx:114-128`
- **Impacto**: Botones sin aria-labels, sin focus rings, sin landmark nav

### C13. `getActiveTemplates()` ignora parametro `category`
- **Archivo**: `src/server/queries/message-templates.ts:14-34`
- **Impacto**: Bug logico — filtra por `isActive` pero descarta `category`

## FORTALEZAS (38) — Lo que esta bien hecho

### Seguridad
- Todas las server actions tienen `"use server"` directive
- Todas verifican sesion antes de ejecutar mutaciones
- SQL injection prevenida via Drizzle ORM parametrizado
- Sin secrets hardcodeados en codigo fuente
- `.env` files en `.gitignore`
- TypeScript strict mode habilitado

### Arquitectura
- Directorio bien organizado (app/components/server/lib/hooks/types)
- Data flow consistente: Server Component -> Query -> Client Component -> TanStack Query -> Server Action
- ActionResult<T> discriminated union en todas las acciones
- State machines puros (data + funciones puras, sin dependencias)
- Transiciones enforced dentro de transacciones DB con row-level lock
- Storage abstracted behind interface (swappable)
- Validators compartidos entre cliente y servidor via Zod

### Base de Datos
- Estructura de tablas consistente (id UUID, createdAt, updatedAt)
- Enums PostgreSQL para data integrity
- Schema namespace `hsm` aislado
- Connection pooling correcto (prepare: false para Supabase)
- Lazy singleton para DB connection
- Foreign keys bien considerados (SET NULL, RESTRICT segun contexto)
- Valores monetarios en bigint (centavos)
- Paginacion bien implementada (Promise.all data + count)

### Frontend
- Separacion server/client components correcta
- DataTable reutilizable y production-ready
- URL state sincronizado con nuqs (shareable, bookmarkable)
- React Hook Form + Zod integration idomatic
- Cascading dropdowns con auto-fill (client -> location -> address)
- Toast notifications consistentes (sonner)
- Error boundary global con recovery UI
- Loading skeletons para paginas principales
- shadcn/ui con full WAI-ARIA en primitivos base

### Testing
- State machines bien testeados (transitions + rejections)
- Incident y RMA validators con tests comprehensivos
- Utils (aging, date-format) con tests
- Server actions de incidents, rmas, users con tests
